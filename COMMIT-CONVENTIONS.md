# Commit conventions I have used

  The structural format of the commit messages I have used is a short standard one

  Format that every commit messages must follow

  <Type of commit>:<Short description of the changes>

# Type of Commits I have used

    Types                 example

    feat                  feat:updated the about page
    fix                   fix : merge of the branches
    docs                  docs: updated the readme about x-y-z
    Hotfix                hotfix: Bugs fixed
    testing               testing: stash work testing

# Good vs Bad commits:

  Good commit:  feat:added the sidebar to index.html

  Bad commit: added sidebar

# Why consistent commit message matters for the team

  The purpose of the commit message is to understand the changes happened in the codebase
  So it is extremely important for commit message to be of single - format that is 
  understood by everyone in team.
